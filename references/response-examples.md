# 输出正反例

这个文档用于约束 `property-advisor` 的最终回答风格。

## 反例：把内部过程直接展示给用户

下面这种输出是不合格的：

```markdown
第一步：硬约束过滤
搜索结果：15 条样本

过滤项 | 标准 | 结果
预算 | <= AUD 10,000/月 | 全部通过
...

第二步：样本内横向比较
最低：...
最高：...
中位数：...

第三步：候选评估
房源 | 价格 | 图片质量 | 分析
...
```

为什么不合格：

- 这是分析过程，不是用户要的最终建议
- 用户需要的是“看哪几套、为什么、风险是什么”
- 这种输出会让人感觉模型在展示自己的工作流，而不是帮用户做决定

为什么合格：

- 先给结论
- 只展示用户需要的部分
- 每条推荐都有理由，也有未确认风险
- 没有把中间分析过程暴露出来

## 反例：地图低置信后输出失去价值

下面这种输出是不合格的：

```markdown
推荐 58 Clarke St、25 Coventry St、70 Southbank Boulevard。
这些房源都在 Southbank，配套不错，交通需要复核。
建议联系房东确认信息。
```

为什么不合格：

- 没解释为什么推荐这几套，用户无法缩小选择
- 没有说明预算是否覆盖
- 没有排除异常价或不匹配房源
- 只说“交通需要复核”，没有告诉用户应该复核什么
- 没有体现 Property Advisor 的决策价值
