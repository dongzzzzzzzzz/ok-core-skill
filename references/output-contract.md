# 最终输出合同

这个文档定义 `property-advisor` 在最终回答阶段必须遵守的展示规则。

核心原则：**内部分析可以复杂，给用户的回答必须简洁、结论导向、可执行。**

## 一、默认展示顺序

默认按下面顺序输出：

1. 一句话结论
2. 优先建议 / 可继续关注的候选
3. 风险提醒
4. 下一步建议

除非用户明确要求“展开分析过程”，否则不要改变这个顺序。

## 二、按模式输出

### 1. 推荐模式

适用于候选较清晰时。

推荐结构：

```markdown
[城市/区域 + 需求 + 预算] 推荐

优先看这 2-3 套：

1. [房源标题]
价格：...
适合你的原因：...
需要确认：...
链接：原帖 ...；地图复核 ...

2. [房源标题]
...

风险提醒：
- ...
- ...

下一步：
- ...
```

硬规则：

- 最多推荐 3 套
- 每套最多 4 行正文
- 可以排序，但不要写 “Top1”“五星”“综合评分 83 分”

### 2. 观察模式

适用于有候选，但还不足以做强推荐时。

推荐结构：

```markdown
当前有几套值得继续看，但还不建议直接下结论。

可继续关注：

1. [房源标题]
目前优点：...
当前缺口：...
链接：原帖 ...；地图复核 ...

风险提醒：
- ...

下一步：
- 优先补查 ...
```

硬规则：

- 不要伪装成已经完成推荐
- 要明确指出“为什么现在还不能直接推荐”

### 3. 仅说明模式

适用于样本很少、数据严重不足、或搜索结果和需求偏差大。

推荐结构：

```markdown
按你当前条件，现有结果还不足以做可靠推荐。

目前能确认的是：
- ...
- ...

当前不能确认的是：
- ...
- ...

建议下一步：
- 放宽/收紧什么条件
- 换哪个数据源或补什么字段
```

硬规则：

- 不要硬凑推荐名单
- 不要为了“看起来有结果”而输出弱数据房源

## 三、每套房源必须包含什么

只要提到某套房源，至少要包含：

- 标题
- 价格
- 原帖链接
- 一条适合用户的理由
- 一条仍需确认的风险或缺失字段

链接规则：

- 原帖链接必须来自房源数据的 `url` 或地图结果的 `listing_ref.url`
- Google Maps / OpenStreetMap 只能作为地图复核链接，不能替代原帖链接
- 如果某套房缺少原帖链接，不能作为强推荐，只能放入“需补链接后再看”
- 使用地图上下文时，链接行应同时区分“原帖”和“地图复核”

如果有图片信息，也应提一句：

- “有实拍图 / 图片较少 / 只有占位图”

如果使用了地图上下文，也必须遵守：

- 地址级结果可以写“基于 OSM 估算，约...”
- 区域级结果只能写“该区域整体...”，不要写到某套房的精确距离
- degraded 或缺失地图结果时，不能输出周边/通勤结论
- 公共 OSM 结果不能写成 Google Maps 已验证
- OSM 未记录到某类设施时，写“未记录到 / 需复核”，不要写“没有”
- 地图 POI 数量表不要原样展示；每套房最多保留 1 条和决策直接相关的地图信号
- 所有距离必须写清楚“从哪里到哪里”和距离类型；禁止写“直线约 1.8km”“tram 约 100m”这种缺主语表达
- 到公共交通必须写出站点名，例如“到 OSM 记录的 Stop 19: Shrine of Remembrance 直线估算约 100-150m”
- `source=photon` 或 `confidence=low` 时，只能作为低置信筛选信号，不能把具体距离当成强推荐理由

## 四、地图低置信时也必须有用

如果地图结果是 `precision=area`、`confidence=low`、`source=photon`、`status=partial` 或 `status=degraded`，不要输出空泛结论。必须改用非地图维度完成初筛：

- 价格：是否在预算内，按当地周期换算，例如澳洲按周租
- 区域：CBD 内、CBD 边缘、Southbank、Docklands、远郊等位置层级
- 房型：是否满足卧室数，混入 2/3 房或买卖房时直接排除
- 异常：明显低价、价格单位不明、学生公寓、合租/床位嫌疑
- 缺口：面积、卫浴、家具、bill、租期、入住时间、真实通勤
- 下一步：让用户先联系哪几套，向房东问什么，用 Google Maps 手动复核什么

如果地图结果是 `precision=missing`，仍要消费 `verification_links.google_maps_manual`：

- 每套候选至少给出可手动验证链接，或说明链接可点开核实
- 明确用户点开后应验证：到工作/学校的路线、最近 tram/train/bus、超市/药店、主路/铁路/施工噪音
- 不要把 `geo_missing` 写成“地图没用”，而是写成“自动定位失败，需要手动验证”

地图低置信时可以写：

- “地图只能做区域级参考，不能确认这套房到站点的步行距离”
- “这套仍值得先联系，理由是价格/区域/房型更匹配，而不是地图距离”
- “如果你给出工作/学校地址，下一轮应按真实通勤重新排序”

地图低置信时禁止写：

- “交通很好”
- “步行 X 分钟到 tram”
- “周边 X 米内有 Y 个超市”
- “Google Maps 显示...”

## 五、禁止输出

默认禁止：

- 分析过程表
- 完整“符合预算 / 超预算 / 周边配套”大表
- 原样平铺地图搜索或 POI 统计结果
- “第一步/第二步/第三步”
- 样本统计过程
- 自检表
- 维度执行情况表
- “我重新按框架分析”
- “我没有做分析，是我的问题”
- 内部字段名，例如 `decision_mode`、`explain_only`
- 未经说明的打分、星级、综合评分

## 六、长度控制

面向用户的默认回答：

- 推荐模式：控制在 12-20 行
- 观察模式：控制在 10-16 行
- 仅说明模式：控制在 8-12 行

如果用户明确要求详细展开，再增加内容。

## 七、语言风格

- 先结论，后理由
- 少讲方法，多讲判断
- 用“适合你 / 不建议 / 还需要确认”这种决策语言
- 避免像分析师 memo

## 八、缺失字段处理

如果数据缺少面积、卫浴、设施、朝向、通勤实测等关键字段，统一按这个规则处理：

- 可以继续初筛
- 不可以假装已经确认
- 必须显式写“仍需确认”

推荐表达：

- “当前更像初筛结果，不是最终结论”
- “面积/卫浴/设施当前没有，需要联系房东或看房确认”
- “通勤时间建议用 Google Maps 在工作日早高峰再核实一次”

地图相关推荐表达：

- “基于 OSM 地址级数据，从 205 Normanby Rd 到 OSM 记录的 Stop 126: Montague Street 直线估算约 150-200m；这不是 Google Maps 步行路线，需复核”
- “当前只有区域级位置，只能判断 Southbank 整体配套较成熟，不能确认这套房楼下周边”
- “公共 OSM 未记录到 800m 内药店，不代表现实中没有，建议看房前复核”
