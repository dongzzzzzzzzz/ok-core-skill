# 最终输出合同

这个文档定义 `property-advisor` 在最终回答阶段必须遵守的展示规则。

核心原则：**内部分析可以复杂，给用户的回答必须简洁、结论导向、可执行。**

## 一、默认展示顺序

默认按下面顺序输出：

1. 一句话结论
2. 优先建议 / 可继续关注的候选
3. 风险提醒
4. 下一步建议

除非用户明确要求“展开分析过程”，否则不要改变这个顺序。

## 二、按模式输出

### 1. 推荐模式

适用于候选较清晰时。

推荐结构：

```markdown
[城市/区域 + 需求 + 预算] 推荐

优先看这 2-3 套：

1. [房源标题]
价格：...
适合你的原因：...
需要确认：...
链接：...

2. [房源标题]
...

风险提醒：
- ...
- ...

下一步：
- ...
```

硬规则：

- 最多推荐 3 套
- 每套最多 4 行正文
- 可以排序，但不要写 “Top1”“五星”“综合评分 83 分”

### 2. 观察模式

适用于有候选，但还不足以做强推荐时。

推荐结构：

```markdown
当前有几套值得继续看，但还不建议直接下结论。

可继续关注：

1. [房源标题]
目前优点：...
当前缺口：...
链接：...

风险提醒：
- ...

下一步：
- 优先补查 ...
```

硬规则：

- 不要伪装成已经完成推荐
- 要明确指出“为什么现在还不能直接推荐”

### 3. 仅说明模式

适用于样本很少、数据严重不足、或搜索结果和需求偏差大。

推荐结构：

```markdown
按你当前条件，现有结果还不足以做可靠推荐。

目前能确认的是：
- ...
- ...

当前不能确认的是：
- ...
- ...

建议下一步：
- 放宽/收紧什么条件
- 换哪个数据源或补什么字段
```

硬规则：

- 不要硬凑推荐名单
- 不要为了“看起来有结果”而输出弱数据房源

## 三、每套房源必须包含什么

只要提到某套房源，至少要包含：

- 标题
- 价格
- 链接
- 一条适合用户的理由
- 一条仍需确认的风险或缺失字段

如果有图片信息，也应提一句：

- “有实拍图 / 图片较少 / 只有占位图”

## 四、禁止输出

默认禁止：

- 分析过程表
- “第一步/第二步/第三步”
- 样本统计过程
- 自检表
- 维度执行情况表
- “我重新按框架分析”
- “我没有做分析，是我的问题”
- 内部字段名，例如 `decision_mode`、`explain_only`
- 未经说明的打分、星级、综合评分

## 五、长度控制

面向用户的默认回答：

- 推荐模式：控制在 12-20 行
- 观察模式：控制在 10-16 行
- 仅说明模式：控制在 8-12 行

如果用户明确要求详细展开，再增加内容。

## 六、语言风格

- 先结论，后理由
- 少讲方法，多讲判断
- 用“适合你 / 不建议 / 还需要确认”这种决策语言
- 避免像分析师 memo

## 七、缺失字段处理

如果数据缺少面积、卫浴、设施、朝向、通勤实测等关键字段，统一按这个规则处理：

- 可以继续初筛
- 不可以假装已经确认
- 必须显式写“仍需确认”

推荐表达：

- “当前更像初筛结果，不是最终结论”
- “面积/卫浴/设施当前没有，需要联系房东或看房确认”
- “通勤时间建议用 Google Maps 在工作日早高峰再核实一次”
